# DATATHON PROJECT MASTER CONTEXT

## Part 3 — Project Constitution, Future Workflow, Phase Philosophy, Team Operating System

---

# PURPOSE OF THIS DOCUMENT

This document acts as the operating system of the project.

Future discussions, prompts, planning, implementation, architecture reviews, and team decisions should follow the principles defined here.

This document exists to prevent:

* Scope creep
* Random feature additions
* Poor planning
* Unstructured coding
* Repeated discussions
* Context loss

---

# PROJECT CONSTITUTION

The project follows a strict rule:

```text
Understand First
Build Later
```

The project does NOT follow:

```text
Build First
Understand Later
```

---

# CORE DEVELOPMENT PHILOSOPHY

Every feature must satisfy:

```text
Problem Statement
↓
Need
↓
Feature
```

NOT:

```text
Interesting Idea
↓
Feature
```

# PROMPT PHILOSOPHY

The project does NOT use prompts like:

```text

```

Instead:

```text

↓
Objective
↓
Constraints
↓
Expected Output
↓
Deliverables
↓
Success Criteria
```

Every prompt should minimize ambiguity.

Every prompt should maximize clarity.

# PHASE PHILOSOPHY

The project will be developed through phases.

The team NEVER attempts:

```text
Build Everything
```

The team ALWAYS follows:

```text
Phase
↓
Review
↓
Next Phase
```

---

# PHASE DESIGN RULES

Every phase must define:

## Objective

What is being achieved?

---

## Inputs

What information is required?

---

## Outputs

What will be produced?

---

## Deliverables

What files, features, or systems will exist?

---

## Success Criteria

How do we know the phase is complete?

---

## Dependencies

What must exist before the phase begins?
